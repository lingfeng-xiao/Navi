# 项目版本管理规范

## 版本号格式

本项目采用语义化版本控制（Semantic Versioning）规范，格式为：

`MAJOR.MINOR.PATCH-PRERELEASE+BUILD`

- **MAJOR**: 当你做了不兼容的 API 修改
- **MINOR**: 当你添加了向下兼容的功能性新增
- **PATCH**: 当你做了向下兼容的问题修正
- **PRERELEASE** (可选): 预发布版本标识符，如 `beta.1`, `alpha.2`, `rc.1`
- **BUILD** (可选): 构建元数据，如构建日期或提交哈希

## 版本发布流程

1. **开发阶段**
   - 在 `main` 分支或特性分支上进行开发
   - 完成功能开发和测试

2. **版本准备**
   - 更新 `package.json` 中的版本号
   - 提交版本更新: `git commit -m "chore: 更新版本号到 X.Y.Z"`

3. **创建标签**
   - 为当前版本创建 Git 标签: `git tag -a vX.Y.Z -m "Release version X.Y.Z"`

4. **发布流程**
   - 推送到远程仓库: `git push origin main --tags`
   - 确保 CI/CD 流程正常执行

5. **文档更新**
   - 更新此文档中的版本历史
   - 更新 README.md 中的相关信息

## 版本管理最佳实践

1. **分支管理**
   - `main`: 主分支，包含生产就绪的代码
   - 特性分支: 用于开发新功能
   - 修复分支: 用于修复生产环境问题

2. **标签管理**
   - 每个发布版本必须创建对应的标签
   - 使用 `v` 前缀命名标签，如 `v1.0.0`
   - 标签必须包含详细的发布说明

3. **版本号更新规则**
   - 非向后兼容的更改必须增加主版本号
   - 新功能且向后兼容的更改必须增加次版本号
   - 向后兼容的错误修复必须增加补丁版本号

4. **预发布版本**
   - 使用 `alpha`、`beta`、`rc`（发布候选）等标识预发布版本
   - 预发布版本用于测试新功能

## 注意事项

- 严禁在代码中硬编码敏感信息（如API密钥、Token等）
- 使用环境变量或配置文件管理敏感信息
- 确保所有依赖项使用固定版本或范围版本控制
- 定期更新依赖项以修复安全漏洞
